# Проект прогнозирования невозврата рассрочек BNPL (Buy Now Pay Later)

## Описание проекта

Проект направлен на решение задачи оперативного прогнозирования невозврата платежей по рассрочкам (сплитам). Система позволяет предсказывать финансовые риски, связанные с возможными просрочками и дефолтами пользователей, оформляющих заказы с оплатой частями (сплиты).

Целью является разработка ML-моделей, обеспечивающих точные прогнозы для следующих задач:

1. **Прогноз величины просроченной задолженности:**

   * на горизонте 1 месяц;
   * на горизонте 2 месяца.

2. **Вероятность наличия просроченной задолженности более 500 руб.:**

   * на горизонте 1 месяц;
   * на горизонте 2 месяца.

3. **Вероятность отсутствия успешных погашений:**

   * на горизонте 1 месяц;
   * на горизонте 2 месяца.

## Описание данных

В папке `data` представлены следующие исходные файлы:

### 1. Основной датасет: `result.csv`

Таблица содержит основную информацию по заказам:

* `order_id` — идентификатор заказа.
* `application_id` — идентификатор пользователя.
* `order_amount` — общая сумма текущего заказа (долг + комиссия).
* `order_dt` — дата создания заказа.
* `split_count` — количество платежей (сплитов).
* дополнительные синтетические признаки, отражающие историю поведения пользователя:

  * `previous_orders_count` — количество предыдущих заказов у пользователя.
  * `split_amount` — сумма одного сплита текущего заказа.
  * `prev_delayed_splits_count` — число просроченных платежей во всех предыдущих заказах.
  * `prev_median_delay_all`, `prev_mean_delay_all`, `prev_std_delay_all` — статистики просрочек всех платежей предыдущих заказов.
  * `prev_median_delay_delayed`, `prev_mean_delay_delayed`, `prev_std_delay_delayed` — статистики просрочек только просроченных платежей.
  * `prev_max_consecutive_delayed` — максимальное число подряд просроченных платежей.
  * `prev_delayed_splits_per` — доля просроченных платежей в предыдущих заказах.
  * `prev_orders_delayed_ratio` — доля предыдущих заказов с просрочками.
  * `order_history_days` — дней от первого заказа пользователя до текущего заказа.
  * `total_amount`, `median_amount`, `mean_amount`, `std_amount` — статистики сумм предыдущих заказов.
  * `median_split_count`, `mean_split_count`, `std_split_count` — статистики количества сплитов предыдущих заказов.
  * `median_split_amount`, `mean_split_amount`, `std_split_amount` — статистики сумм сплитов предыдущих заказов.
  * `open_orders_count` — число открытых заказов на момент текущего заказа.
  * `debt_sum` — суммарный долг по открытым заказам на момент текущего заказа.
  * `mean_delay_all` — медианное число дней просрочки по текущему заказу.
  * `first_delayed_split_num` — номер первого просроченного сплита текущего заказа.
  * `delayed_splits_per` — доля просроченных платежей текущего заказа.

### 2. Расписание платежей: `schedule_before_day_second_payment_razum_ai.xlsx`

Основные колонки:

* `order_id` — идентификатор заказа.
* `plan_payment_dt` — плановая дата платежа.
* `debt` — сумма планового платежа.

Эти данные используются для расчета временных интервалов и определения плановых дат погашения.

### 3. Транзакционные данные: `refill_with_error_codes_razum_ai1.xlsx` и `refill_with_error_codes_razum_ai2.xlsx`

Основные колонки:

* `transaction_dt` — дата транзакции.
* `transaction_id` — идентификатор транзакции.
* `order_id` — идентификатор заказа.
* `application_id` — идентификатор пользователя.
* `status` — статус транзакции (`success` – успешный платеж, иное – неуспешный).

Эти данные используются для определения фактических платежей и их успешности.

## Целевые признаки для генерации LLM-агентом

На основании имеющихся данных LLM-агенту предстоит сформировать три новых целевых признака в таблице `results.csv`:

### Целевые признаки:

1. **`overdue_amount_1m` и `overdue_amount_2m`**:

   * сумма просроченной задолженности, накопленная в течение 1 и 2 месяцев после даты заказа.

2. **`overdue_500r_prob_1m` и `overdue_500r_prob_2m`**:

   * бинарный индикатор или вероятность того, что сумма просроченной задолженности превысит 500 руб. через 1 и 2 месяца после даты заказа.

3. **`no_successful_payment_prob_1m` и `no_successful_payment_prob_2m`**:

   * бинарный индикатор или вероятность того, что не произойдет ни одного успешного погашения в течение 1 и 2 месяцев после даты заказа.

## Методология формирования целевых признаков

Для корректного формирования целевых признаков:

* Использовать данные о расписании платежей для определения плановых дат.
* Использовать транзакционные данные для расчета реальных оплат и их статусов.
* Сравнивать запланированные платежи с фактическими, выявляя просроченные суммы и сроки просрочки.
* Расчёты необходимо выполнять относительно даты заказа для горизонтов в 1 и 2 месяца.
